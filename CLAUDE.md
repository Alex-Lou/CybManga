# SumiWire Pro — Guide de contribution (canonique)

Éditeur professionnel de manga / comics. Monorepo : **frontend** (React + Electron) et
**backend** (Spring Boot). Ce fichier est la source de vérité des conventions du dépôt.
Toute contribution — humaine ou assistée — s'y conforme.

---

## 1. Architecture (ce qui existe, à respecter)

```
manga-studio-pro/
├── frontend/                     React 18 · Vite 5 · Tailwind 3 · Electron 40
│   ├── src/
│   │   ├── components/<feature>/  UI par domaine (canvas, drawing, toolbar, bubbles, panels…)
│   │   ├── context/studio/        état global : state.js · actions.js · reducer.js (reducer PUR)
│   │   ├── context/StudioContext.jsx  provider + autosave (30 s) + fallback localStorage
│   │   ├── styles/                main.css (CSS statique + variables) · tailwind.js (classes centralisées)
│   │   └── utils/                 api.js (SEUL point réseau → /api/v1) · helpers · constants · psd* · …
│   └── electron/main.cjs          shell desktop
└── backend/                       Spring Boot 3.2 · Java 21 · H2 (fichier) / PostgreSQL
    └── src/main/java/com/mangastudio/
        ├── controller/  service/  repository/  entity/  config/  exception/
        └── (desktop : BrowserLauncher, ModernSplashScreen, StartupDisplay)
```

**Invariants à ne pas casser** (le socle sain) :
- L'état global vit dans le **reducer pur** `context/studio/reducer.js`. Pas d'effet de bord dedans.
- Tout appel réseau passe par `utils/api.js`. Aucun `fetch` ailleurs.
- La persistance passe par la liste unique `PERSISTABLE_KEYS` (`utils/helpers.js`).
- Le backend ne sert l'UI que via la couche `controller`. Les règles métier vivent dans `service`.

---

## 2. Règles non négociables

1. **Demander, ne pas supposer.** Ambiguïté → on pose la question avant d'écrire du code.
2. **La solution la plus simple d'abord.** KISS · YAGNI. Pas d'abstraction non demandée.
3. **Ne pas toucher au code non lié.** On ne renomme/réorganise/reformate rien hors scope.
   Une remarque hors sujet se signale en note de fin, on ne la corrige pas en passant.
4. **Confirmation explicite** avant : supprimer un fichier, réécrire du contenu existant,
   migration/schéma DB, appel externe, push/déploiement. Une autorisation passée ne vaut pas
   pour le message courant.
5. **Taille des fichiers : 400 lignes max.** Plafond dur **600** si vraiment impossible de
   découper proprement. Au-delà → extraire (hook `use*.js`, helper pur, sous-composant, module).
6. **Aucune trace d'outil IA** : pas de `Co-Authored-By`, pas de mention « generated by »,
   pas de commentaire signé, ni dans le code ni dans les messages de commit.

---

## 3. Principes de code (appliqués par défaut)

- **DRY** (avec prudence : ne pas factoriser deux codes qui vont diverger) · **SRP** · **SoC** ·
  **Fail fast** (valider/asserter tôt, retours explicites) · lisible > malin.
- **Séparation des responsabilités** : UI / logique métier / état / appels API / persistance / config
  restent distincts. Composition > grosses hiérarchies. Modules petits, cohérents, testables.
- **Règle de décision** (plusieurs solutions) dans l'ordre : 1) la plus sûre, 2) la plus simple,
  3) la plus lisible, 4) la plus testable, 5) la plus facile à faire évoluer.

---

## 4. Frontend

- **Source de vérité unique de l'état** : le reducer. Pas de state local dupliquant l'état global,
  pas de store parallèle. L'état de session (transform, rotation, masque d'édition…) est reset au chargement.
- **Design tokens — UNE source.** Les couleurs runtime vivent dans `THEME_COLORS` (`utils/constants.js`,
  consommé via `theme.*` en inline-style). Les variables CSS de `main.css` servent le CSS **statique**
  (scrollbars, sliders, animations). Ces deux jeux **doivent rester synchronisés** (idéalement les
  inline-styles dérivent des variables CSS). **Interdit** : classe Tailwind de couleur en dur
  (`text-blue-500`, `bg-red-50`…) — elle casse le dark mode. Couleur = `theme.*` ; structure = Tailwind.
- **Tailwind centralisé** dans `styles/tailwind.js`. On n'invente pas de classes utilitaires ailleurs.
- **Composants petits, réutilisables.** La logique lourde (handlers, géométrie, moteur de rendu)
  sort en hook (`use*.js`) ou util pur — cf. le pattern `useBubbleHandlers.js` (référence à suivre).
- **États d'UI complets** : loading / empty / error / success. Actions optimistes avec rollback si l'API échoue.
- **a11y** : navigation clavier, contrastes, focus visibles, responsive.

---

## 5. Backend

- **Layering strict** : `controller` (HTTP/validation) → `service` (métier) → `repository` (accès données).
  L'entité JPA ne porte pas de logique métier.
- **Jamais exposer l'entité.** Les endpoints acceptent/retournent des **DTO** ; le mapping se fait
  dans le service. Le client ne fixe jamais `id` / `createdAt` / `updatedAt`.
- **Validation côté serveur obligatoire** (jakarta : `@Valid`, `@NotBlank`, `@Size`…), même si le front valide.
- **Listes bornées** : pagination (`Pageable`), filtres, limites. Pas de `findAll()` non borné exposé.
- **Erreurs stables** : exceptions métier dédiées mappées au bon statut. Une `RuntimeException`
  générique → 500, pas 404. Ne pas fuiter de message interne au client en prod (`include-message: never`).
- **Authentification ≠ autorisation.** Distinguer « qui es-tu » et « as-tu le droit ».
- **Sécurité** : console H2 et `web-allow-others` **désactivées hors dev local** ; mot de passe DB non vide ;
  CORS externalisé (pas en dur) ; ne jamais logger secrets/tokens ; rate-limit sur endpoints sensibles.

---

## 6. Données & DB

- Évolution du schéma par **migration** explicite. Intégrité via contraintes / transactions / validations.
- Indexer les champs de recherche / jointure / tri fréquents. Gérer concurrence, retries, échecs partiels.

---

## 7. Tests & qualité

- Tests sur la **logique métier critique** (reducer, services). Test de régression pour chaque bug important.
- Pyramide : beaucoup d'unitaires, quelques intégrations, peu d'E2E.
- Un changement n'est correct que s'il est **lisible, testé quand pertinent, et cohérent avec l'archi**.
- Avant de livrer : build vert, lint/format OK, comportement vérifié.

---

## 8. Commandes

```bash
# Frontend
cd frontend
npm install
npm run dev            # Vite dev → http://localhost:3000 (proxy /api → :8080)
npm run build          # bundle → frontend/dist/
npm run electron:dev   # build + lance Electron
npm run electron:build # installeur Windows portable → frontend/release/

# Backend
cd backend
mvn spring-boot:run    # API Spring Boot → http://localhost:8080
mvn package            # jar (copie frontend/dist → static/)
```

---

## 9. Git

- Messages de commit : **impératif, concis, en français**, scope clair (ex. `canvas: extrait le pan/zoom en hook`).
- **Aucun trailer `Co-Authored-By` ni mention d'outil IA.**
- Ne jamais committer d'artefacts (`node_modules`, `dist`, `release`, `target`, sorties jpackage,
  `*.mv.db`) — cf. `.gitignore`. Ne jamais committer de secret.
- Branche dédiée par sujet ; pas de push sans validation explicite.
